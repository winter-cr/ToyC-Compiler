# ToyC 编译器六天开发任务与四人分工

## 目标与范围

本项目目标是在六天内完成一个 ToyC 到 RISC-V32 汇编的编译器。编译器从标准输入读取 ToyC 源码，向标准输出写出 RISC-V32 汇编，并支持在性能测试中接收 `-opt` 参数。实现重点依次为：功能正确、接口符合评测要求、基础性能优化、实践报告完整。

默认技术路线采用 C++20 + CMake。若团队更熟悉 Java，也可保持同样模块划分切换到 JDK 21 + Maven/Gradle，但六天周期内不建议中途更换技术栈。

## 模块划分

1. 词法分析与语法分析
   - 识别关键字、标识符、整数、运算符、分隔符、注释和空白。
   - 解析 `CompUnit`、声明、函数定义、语句、表达式、函数调用。
   - 生成 AST，并保证运算符优先级、结合性和短路表达式结构正确。

2. 语义分析
   - 建立全局/局部/嵌套作用域符号表。
   - 检查变量、常量、形参、函数定义和函数调用。
   - 支持常量表达式编译期求值。
   - 检查 `main`、返回路径、`break/continue` 位置、`void` 函数调用使用限制。

3. 中间表示与代码生成
   - 将 AST 降低到便于生成汇编的 IR 或三地址码。
   - 生成 RISC-V32 汇编，覆盖全局变量、局部变量、函数调用、控制流、短路求值和算术/逻辑/关系运算。
   - 维护栈帧、参数传递、返回值、临时值和标签。

4. 优化、测试与报告
   - 在 `-opt` 下启用常量折叠、死代码删除、简单公共子表达式消除或局部寄存器分配。
   - 建立本地测试集，对比 ToyC 源码用 C 编译运行后的退出码。
   - 编写实践报告，说明架构、关键算法、测试与优化效果。

## 四人分工

| 成员 | 主要职责 | 交付物 |
| --- | --- | --- |
| A：前端负责人 | 词法分析、递归下降/ANTLR 语法分析、AST 定义与构建 | Lexer、Parser、AST 节点、语法单元测试 |
| B：语义负责人 | 符号表、作用域、类型/常量/返回检查、错误诊断 | SemanticAnalyzer、SymbolTable、常量求值、语义测试 |
| C：后端负责人 | IR 设计、RISC-V32 代码生成、栈帧和调用约定 | IR、CodeGen、汇编输出、后端测试样例 |
| D：测试与优化负责人 | 自动化测试、性能优化、报告和集成 | 测试脚本、优化 Pass、实践报告、CI/构建检查 |

每位成员需要为自己负责模块提供最小可运行样例和接口文档。每天结束前必须合并一次可构建版本，避免最后一天集中集成。

## Git 分支与提交管理

`main` 分支只保存已经集成、可构建、可运行的版本，作为最终评测和阶段验收使用的稳定分支。任何成员不得直接向 `main` 分支提交或强制推送代码。

每位成员在自己的功能分支上开发，并将分支推送到远程仓库。建议分支命名如下：

| 成员 | 分支名 | 负责范围 |
| --- | --- | --- |
| A | `feature/frontend` | 词法分析、语法分析、AST |
| B | `feature/semantic` | 符号表、语义分析、常量求值 |
| C | `feature/backend` | IR、RISC-V32 代码生成 |
| D | `feature/test-opt-report` | 测试、优化、报告、集成 |

提交和合并规则：

- 成员只在自己的分支上提交代码，并使用 `git push origin <branch>` 推送。
- 每次准备合并前，成员需要先从 `main` 同步最新代码到自己的分支，解决本分支内的冲突后再提交合并请求。
- D 作为集成负责人，统一审核并合并各成员分支到 `main`。
- 合并到 `main` 前必须至少通过项目构建和本模块相关测试。
- 如果某个分支会影响公共接口，例如 AST 节点、符号表结构、IR 格式或代码生成接口，合并前必须在群内同步变更点。
- `main` 出现构建失败时，优先修复 `main`，暂停继续合并新功能分支。
- 禁止在 `main` 上提交未完成的临时代码、调试输出、个人测试路径或无法通过构建的改动。

推荐日常流程：

```bash
git checkout main
git pull origin main
git checkout feature/frontend
git merge main
# 本地开发、构建、测试
git add <changed-files>
git commit -m "feat(frontend): add expression parser"
git push origin feature/frontend
```

## 六天开发计划

### 第 1 天：项目骨架与最小前端

目标：确定技术栈和目录结构，完成可读取输入、解析最简单程序的编译器骨架。

- A：定义 Token、Lexer、AST 基类和表达式/语句节点；完成关键字、标识符、整数、注释、运算符扫描。
- B：设计符号表接口和语义分析入口；整理任务要求中的语义约束清单。
- C：确定 IR 或直接代码生成接口；输出最小 RISC-V 汇编模板。
- D：搭建 CMake、测试目录和样例运行脚本；准备 10 个基础 ToyC 样例。

里程碑：
- `compiler < tests/basic/return_0.tc > out.s` 能输出汇编框架。
- 项目可通过 `cmake --build build` 构建。

### 第 2 天：完整语法与 AST

目标：覆盖 ToyC 全部语法，能构建完整 AST。

- A：完成声明、函数定义、语句块、if/else、while、break、continue、return、赋值、函数调用解析。
- B：接入 AST 遍历框架，开始实现全局符号收集。
- C：实现表达式和语句 IR 草案，明确短路逻辑和标签表示。
- D：扩充语法测试，覆盖运算符优先级、嵌套作用域、函数调用和控制流。

里程碑：
- 所有合法语法样例能解析成 AST。
- 表达式优先级与 C 语言结果一致。

### 第 3 天：语义分析闭环

目标：完成大部分语义检查，使后端可以假设输入 AST 已合法。

- A：配合 B 修复 AST 信息缺口，例如源码位置、节点类型、声明属性。
- B：完成作用域、变量/常量声明、函数唯一性、调用参数、常量表达式、返回语句、循环上下文检查。
- C：实现局部变量、全局变量、常量引用和基础表达式的代码生成。
- D：建立语义测试集，包括合法样例和本地调试用非法样例；整理报告中的语义分析章节。

里程碑：
- 合法测试程序能通过语义分析。
- `int main(){ return 1 + 2 * 3; }` 生成汇编运行退出码为 7。

### 第 4 天：控制流、函数调用与全局数据

目标：完成主要功能测试所需的代码生成。

- A：修复前端在复杂嵌套程序中的解析问题。
- B：补齐 `void` 函数调用使用限制、每条路径返回值检查、声明后使用规则。
- C：完成 if/else、while、break、continue、短路 `&&`/`||`、函数调用、参数传递、递归、全局变量读写。
- D：编写端到端测试脚本：ToyC 生成汇编，汇编链接运行，对比 C 编译结果退出码。

里程碑：
- 阶乘、斐波那契、嵌套循环、递归、全局变量样例全部通过。
- 评测接口保持 stdin/stdout，不依赖固定输入文件路径。

### 第 5 天：优化与稳定性

目标：在保证正确性的前提下提高性能测试表现。

- A：处理语法边界问题和注释/负数字面量等词法细节。
- B：为优化提供常量信息和不可变符号信息；复查所有任务要求语义点。
- C：改进后端临时值分配，减少不必要的栈读写；实现基础寄存器复用。
- D：实现或集成优化 Pass：常量折叠、死代码删除、简单跳转合并；运行回归测试并记录优化效果。

里程碑：
- 不带 `-opt` 和带 `-opt` 均能正常输出汇编。
- 基础回归测试全部通过，性能样例无明显超时。

### 第 6 天：收尾、报告与提交

目标：完成最终验证、实践报告和仓库整理。

- A：整理前端设计说明、文法覆盖情况和已知限制。
- B：整理语义规则实现说明，补充符号表与常量求值文档。
- C：整理后端栈帧、调用约定、RISC-V 指令选择和优化前后对比。
- D：完成实践报告汇总、README 检查、构建脚本验证和提交材料准备。

里程碑：
- 全量本地测试通过。
- README、构建配置、实践报告和源码均已提交。
- 仓库可被评测系统克隆并自动构建。

## 每日协作节奏

- 每天开始：15 分钟同步当天目标、接口变更和风险。
- 每天下午：成员将当天可运行改动推送到各自分支，由 D 选择稳定改动合并到 `main`，确保 `main` 始终可构建。
- 每天结束：记录已通过样例、失败样例、次日优先级。
- 任何公共接口变更必须同步到 `docs/` 或 README 中，避免模块间猜接口。
- 注意个人需要在本地维护自己开发时遇到的bug内容和解决记录便于报告(若有)的编写

## 测试策略

1. 单元测试
   - Lexer：关键字、标识符、数字、注释、运算符。
   - Parser：表达式优先级、语句嵌套、函数定义。
   - Semantic：作用域、常量、函数调用、返回路径。

2. 端到端测试
   - 用 ToyC 编译器生成 RISC-V32 汇编。
   - 将同一 ToyC 源码作为 C 代码用 GCC 编译运行，记录退出码。
   - 运行生成汇编对应程序，比较退出码。

3. 回归测试
   - 每修复一个 bug 新增一个最小样例。
   - 第 5 天后不再大改架构，只做局部修复和优化。

## 主要风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 后端栈帧和调用约定实现过晚 | 函数调用、递归、局部变量全部受阻 | 第 3 天开始实现最小函数调用，第 4 天完成递归样例 |
| 语义检查与代码生成耦合过深 | 后期难以定位错误 | AST、Semantic、IR/CodeGen 分层，后端默认输入合法 |
| 优化破坏正确性 | 功能测试失分 | `-opt` 下优化可逐项开关，所有优化必须跑回归测试 |
| 实践报告滞后 | 报告分丢失 | 每天由 D 收集模块说明，第 6 天只做整合和润色 |
